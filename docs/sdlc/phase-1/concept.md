## 目的

温泉・銭湯・サウナを地図で探す Flutter アプリ **YuMap** を、リリース直前段階で「**安心して使える / 触り心地が良い / 動作が確実 / 入口が広い**」状態に底上げする。新規大型機能の追加ではなく、**既存実装の信頼性と完成度を引き上げる仕上げイニシアチブ**である。

解決する問題：
- ユーザーが投稿する写真・レビューが信頼できない（モデレーションなし）
- 主要画面の触り心地と動線にまだ粗さがある
- リリース前にボタン動作が網羅テストされていない（リグレッション検出機構なし）
- ログイン手段がメール/パスワードのみで、新規ユーザー獲得障壁が高い／App Store 審査リスクあり

---

## ターゲットユーザー

| 層 | 想定 |
|---|---|
| 主ユーザー | 温泉・銭湯・サウナを日常的に利用する 20〜60代の一般スマホユーザー |
| 副ユーザー（貢献者） | レビュー・写真・チェックインを投稿する常連層 |
| プラットフォーム | iOS / Android 両方リリース前提（iOS は Sign in with Apple 審査要件あり） |

## スコープ

### やること（4テーマ）

#### (1) 情報の信憑性向上
- **写真モデレーション**: 自動方式（Supabase Edge Function 経由で外部画像モデレーション API、または同等の AI モデル）。投稿時に NSFW / 暴力 / 不適切画像を自動弾き、PASS したものだけ DB 反映。
- **チェックイン距離バリデーション**: 既存ロジックのバイパス可能性を塞ぐ（クライアント側だけでなくサーバー側 RPC でも GPS 距離検証）。
- **不正投稿対策**: スパム/誹謗中傷の文言検出（テキストモデレーション、NGワード辞書 + AI 判定）と、他ユーザーへの嫌がらせ対策（同一ユーザーへの連続レビュー制限・通報フロー）。

#### (2) UI改善
- マーカー（クラスタリング表示崩れ・色分け視認性）
- ボトムシート（高さ・スワイプ・写真カルーセル）
- 検索（地図移動連動・リアルタイム検索の応答性）
- フィード（無限スクロール・施設絞り込み・読み込み感）

#### (3) 各ボタンの動作確認
**実施順序を明確化**（網羅範囲が広いため優先順位順に進める）：
1. **必須パス**: ログイン → 地図 → 施設詳細 → レビュー投稿 → チェックイン → お気に入り → プロフィール
2. **設定系**: 通知設定、アカウント削除、ログアウト
3. **管理系**: 投稿編集・削除、レビュー削除、写真削除

各ボタンに対して「期待動作 / 実動作 / PASS or FAIL / 修正コミット」を記録するチェックリストを `docs/sdlc/phase-X/button-test-matrix.md`（後フェーズで作成）に蓄積する。

#### (4) 認証強化
- **Google ログイン追加**（Android / iOS 両対応）
- **Apple ログイン追加**（iOS は審査要件のため必須、Android でも可能なら対応）
- 既存のメール/パスワード認証は維持（並列方式）
- Supabase Auth の Provider 設定 + Flutter 側 SDK 実装 + プライバシーポリシー / 利用規約のリンク追加

### やらないこと（明示的に除外）
- 新規大型機能（課金、施設データ自動収集、プラン共有の作り直し、店舗側管理画面の刷新など）
- 既存テーブル構造の大規模変更（Phase 3 で必要があれば DB 変更を検討するが、初期スコープには含めない）
- 既存の他プロバイダ認証（Twitter / Facebook など）の追加
- 多言語対応（現状日本語のみで進める）

## 技術的制約

| 項目 | 内容 |
|---|---|
| FE | Flutter（既存 lib/ 配下の実装を改修ベースで使う、Riverpod 状態管理） |
| BE | Supabase（PostgreSQL + RLS 設定済み）、Edge Functions（モデレーション用に新規追加） |
| 地図 | flutter_map（OpenStreetMap）。Google Maps への戻し不可（過去にコスト理由で除去済み） |
| 認証 | Supabase Auth（Google / Apple Provider を有効化） |
| 開発体制 | 1人開発、プログラミング未経験＋AI駆動開発スタイル |
| 安全性 | **既存実装を壊さない**ことが最優先。改修系の変更は機能フラグ or 段階リリースで保護 |
| リリース | iOS / Android 両方。Apple ログインは Sign in with Apple ガイドライン審査要件 |
| 外部 API | 画像モデレーション API（候補: AWS Rekognition / Google Vision SafeSearch / OpenAI Moderation 等、Phase 2 で確定）、テキストモデレーション API（同上） |

## 成功基準（リリース可否ゲート）

| # | 基準 | 検証方法 |
|---|---|---|
| 1 | レビュー/写真の不正混入が自動で抑制される | モデレーション API が稼働、テストデータで NSFW / スパムが弾かれる |
| 2 | チェックインがサーバー側 GPS 検証を通る | クライアント改竄を試みても拒否されることをテスト |

| 5 | Google/Apple 両ログインで新規登録〜投稿まで完走 | 実機テスト（iOS/Android）で完走確認 |
| 6 | リリースチェックリスト全項目クリア | `release-checklist.md`（既存 or 新規）でグリーン |

## 未解決の課題（Phase 2 で解消すべき不確実性）

1. **画像/テキストモデレーション API の選定**
   - 候補: AWS Rekognition / Google Vision SafeSearch / OpenAI Moderation / Hive / Sightengine / Perspective API
   - 評価軸: コスト（無料枠）/ 日本語対応の精度 / Supabase Edge Function からの呼び出しやすさ
2. **不正投稿の通報フロー UI**
   - 「通報ボタン」を施設詳細・レビュー詳細・プロフィールどこに置くか、通報後の処理フロー（管理者通知方法）
3. **Apple Developer Program のステータス確認**
   - Sign in with Apple は有料 Apple Developer Program が必要（年 $99）。既加入か未加入か
4. **Google ログインの bundle ID / SHA-1 設定**
   - 既存の Firebase / GCP プロジェクト有無、未準備なら Phase 3 で構成
5. **ボタン網羅テストの「画面一覧」の確定**
   - 現状の画面数とそれぞれのボタン数を Phase 2 で棚卸しする

## Phase 2 への引き継ぎ事項

### 確定した要件
- 4テーマ並列（信憑性 / UI / ボタン網羅 / 認証強化）
- モデレーションは**自動方式**
- 不正対策は (b) スパム/誹謗中傷 + (c) 嫌がらせ
- Google/Apple **両方**実装、iOS リリースのため Apple は必須
- ボタン確認は**必須パス → 設定系 → 管理系**の順で全部実行
- 既存テーブル構造の大規模変更はしない

### 要確認事項（Phase 2 で確定）
- モデレーション API の選定（候補比較表を Phase 2 で作る）
- 通報フローの UI/UX 配置
- Apple Developer Program の加入状況
- 既存 GCP / Firebase プロジェクトの有無
- 画面・ボタンの全棚卸し（テストマトリクスの母集団確定）

### 制約で最も注意すべき点
- **既存実装を壊さない**こと。セッション35 までで積み上げた機能（チェックイン / お気に入り / プラン / フィード / 写真カルーセル等）にリグレッションを発生させてはいけない
- Apple ログインを入れずに Google ログインだけ実装すると **App Store 審査でリジェクト**される（Sign in with Apple ガイドライン）。必ずセットで実装する
- リリース直前のため、**機能追加 < 安定化** を優先する判断軸を Phase 2 以降でも貫く
