## 0. 着手順序（Phase 1 採点改善点①の解消）

4テーマは並列ではなく、**依存関係に従って Wave 制で順次着手**する：

| Wave | テーマ | 理由 |
|---|---|---|
| **Wave 1** | (4) 認証強化（Google / Apple） | 後続のテストで「ログインしてからの動作」を検証するため、最初に基盤を整える必要がある。Apple Developer Program 加入の有無確認も同時に進める |
| **Wave 2** | (1) 信憑性向上（モデレーション・GPS検証・通報） | 投稿系の安全性は他機能の前提。Wave 3 のUI改善で投稿系UIに触る前に確定させる |
| **Wave 3** | (2) UI改善（マーカー・シート・検索・フィード） | 信頼性が担保されたあとに触り心地を磨く |
| **Wave 4** | (3) ボタン動作確認（必須→設定→管理） | 全変更が入った状態で網羅テストするのが最も意味がある（早期実施だと再テストになる） |

各 Wave は独立してリリース判断可能（途中で止めても前 Wave の成果は活きる）。

## 1. 機能要件（FR）

## 2. 非機能要件（NFR）

Phase 1 採点改善点③の解消。レイテンシ・タイムアウト・フォールバック戦略を明文化。

| ID | 種別 | 要件 | 測定方法 |
|---|---|---|---|
| NFR-001 | パフォーマンス | 写真モデレーション API: P95 < 2秒 | Edge Function ログの所要時間集計 |
| NFR-002 | パフォーマンス | テキストモデレーション API: P95 < 1秒 | 同上 |
| NFR-003 | パフォーマンス | モデレーション API タイムアウト: 5秒 | コード設定値 |
| NFR-004 | 可用性 | モデレーション API 障害時のフォールバック方針: **拒否（投稿不可）** | 障害シミュレーションテスト |
| NFR-005 | パフォーマンス | チェックイン RPC: P95 < 500ms | Supabase logs |
| NFR-006 | パフォーマンス | チェックイン GPS 距離許容: 設定可（デフォルト 100m） | 設定ファイル |
| NFR-007 | パフォーマンス | ログイン処理: P95 < 3秒 | 計測ログ |
| NFR-008 | パフォーマンス | マーカー描画: 1000件で 60fps 維持 | Flutter DevTools |
| NFR-009 | パフォーマンス | 検索リアルタイム応答: 入力後 300ms 以内に候補更新 | UI計測 |
| NFR-010 | セキュリティ | RLS 設定の継続的維持（変更時はマイグレーションで明示） | Supabase Advisors |
| NFR-011 | セキュリティ | OWASP Top 10 準拠（特に A01: Broken Access Control） | 手動チェック |
| NFR-012 | セキュリティ | チェックイン距離検証はサーバー側必須（クライアント検証は補助） | コードレビュー |
| NFR-013 | 互換性 | iOS / Android 両対応 | 両 OS 実機テスト |
| NFR-014 | 互換性 | 既存実装のリグレッションゼロ | Wave 4 のテストマトリクスで全 PASS |
| NFR-015 | 運用性 | モデレーション拒否時のユーザーメッセージ表示 | UI実装 |
| NFR-016 | 運用性 | 通報後の管理者通知（メール/Slack/Supabase ダッシュボード のいずれか） | 通知到達確認 |
| NFR-017 | 開発リソース | 1人開発・AI駆動で 2〜4週間以内完了 | Wave別の進捗 |
| NFR-018 | 法令遵守 | 利用規約・プライバシーポリシーをサインアップ前に提示 | UI実装 |

## 4. 受入テスト骨子（Given-When-Then）

代表的なシナリオ。詳細は Phase 4（ユースケース定義）で展開。

### Wave 1: 認証

| ID | Given | When | Then |
|---|---|---|---|
| AT-001 | アプリを未ログイン状態で起動した | ログイン画面で「Google でログイン」をタップ | Google アカウント選択 → 同意 → ホームに遷移 |
| AT-002 | iOS シミュレータで未ログイン | 「Apple でログイン」をタップ | Apple ID 認証 → ホームに遷移 |
| AT-003 | 既存メールユーザーがログイン中 | 「Google を連携」をタップ | 同一アカウントに Google プロバイダが追加される |
| AT-004 | 規約未同意 | 「サインアップ」を押す | 規約・ポリシーへのリンクが表示され、未同意ならボタンが押せない |

### Wave 2: 信憑性

| ID | Given | When | Then |
|---|---|---|---|
| AT-010 | 不適切な写真（NSFW）をアップロードしようとした | レビューに添付して投稿ボタンを押す | モデレーション API が拒否 → ユーザーに「不適切と判定されました」を表示、DB 反映なし |
| AT-011 | モデレーション API 障害中 | 写真を投稿しようとする | 投稿は拒否され、「現在モデレーション機能停止中です。後ほど再度お試しください」を表示 |
| AT-012 | 施設から 200m 離れた地点でチェックイン試行 | チェックインボタンを押す | サーバー RPC が距離超過と判定 → 拒否 |
| AT-013 | スパム文面のレビュー（NGワード含む） | 投稿ボタンを押す | 拒否され、「不適切な内容が含まれます」を表示 |
| AT-014 | 嫌がらせ的レビューを発見 | 通報ボタン → 理由選択 → 送信 | 通報レコード作成 + 管理者通知 |
| AT-015 | 同一施設に 1時間以内に 2回目のレビュー試行 | 投稿ボタンを押す | 制限により拒否され、残り時間を表示 |

### Wave 3: UI改善

| ID | Given | When | Then |
|---|---|---|---|
| AT-020 | 1000件の施設マーカーがある地図 | ピンチでズームイン/アウト | クラスタリングが崩れず再描画され、60fps を維持 |
| AT-021 | 施設詳細を表示 | ボトムシートをスワイプアップ | スムーズに最大高さに展開 |
| AT-022 | 検索バーに文字入力 | 100ms ごとにキー入力 | 300ms 以内に候補リストが更新される |
| AT-023 | フィード末尾までスクロール | さらに下にスクロール | 次ページがスムーズに読み込まれる |

### Wave 4: ボタン網羅

| ID | Given | When | Then |
|---|---|---|---|
| AT-030 | 必須パスの全画面 | 各画面の主要ボタンを順次タップ | 全ボタンが期待動作 → matrix に PASS 記録 |
| AT-031 | ログアウトボタンを押す | 確認ダイアログで Yes | ログイン画面に戻り、再ログインで元状態復帰 |
| AT-032 | アカウント削除を押す | 確認 → パスワード入力 → 削除 | 削除完了後、未ログイン画面に遷移、再ログイン不可 |

## 5. スコープ外（やらないこと）

- Twitter / Facebook ログイン（要望なし）
- 多言語対応（日本語のみで継続）
- 課金・サブスクリプション
- 施設データの自動収集機能の刷新
- 店舗側管理画面の刷新
- 大規模な DB スキーマ変更（マイグレーションは最小限に留める）

## 6. 制約・前提

### 制約
- リリース直前のため、**機能追加 < 安定化** が判断軸
- 1人開発・未経験ユーザー＋AI駆動のためコード品質はAI生成依存
- 既存実装にリグレッションを発生させない（Wave 4 で全件確認）
- Apple Sign in はiOSリリース必須（年 $99 Apple Developer Program 加入が前提）

### 前提
- Supabase の Auth プロバイダ設定権限がある
- GCP Console / Apple Developer Console の管理権限がある
- リリース時期は Wave 4 完了後

## 7. Phase 3 への引き継ぎ事項

### 設計が必要な領域
- **モデレーション API の選定確定**: 候補は AWS Rekognition / Google Vision SafeSearch / OpenAI Moderation / Hive / Sightengine。Phase 3 で比較表を作って1つに絞る
- **通報フロー設計**: 通報レコードのスキーマ・管理者通知方法（メール？Slack？Supabase Webhook？）
- **チェックイン RPC の SQL 設計**: PostGIS 拡張を使うか、自前で earth_distance を使うか
- **認証フロー設計**: Google/Apple それぞれの OAuth リダイレクト URL・bundle ID 設計
- **アカウントリンク仕様**: 既存メールユーザーが Google を後付けする際の競合解決ロジック

### 環境変数・設定
- 各モデレーション API の API キー管理（Supabase Secrets）
- Google OAuth Client ID（iOS / Android / Web）
- Apple Services ID

### 監視
- 通報数のダッシュボード設計
- モデレーション拒否率のダッシュボード設計
