## 1. 実装不可能 / 矛盾している要件

### 🔴 Issue G-001: NFR-008 のマーカー描画 60fps が達成困難
- **問題**: 「マーカー1000件で 60fps 維持」は flutter_map（OpenStreetMap）+ クラスタリング前提で**かなり厳しい**。実機（特に Android 中位機）で OSM タイルレンダリングと並行すると 30〜45fps に落ちることが多い
- **根拠**: flutter_map は CPU レンダリングベースで、Google Maps SDK のような GPU パイプラインを持たない
- **改善提案**: NFR-008 を「**ズーム時に 1秒以内に再描画完了 / スクロール中 30fps 以上**」に緩和する。または「マーカー描画はクラスタリングで**画面内 100件以下**に抑える」設計制約に書き換える

### 🟡 Issue G-002: アカウントリンク（FR-005）はリリース直前スコープに重い
- **問題**: 既存メールユーザーが Google/Apple を後付けする機能は、Supabase Auth の制約上、**同一メールアドレスで別プロバイダを後付けできない**ケースがある（identities テーブルの制約）
- **根拠**: Supabase の Identity Linking は 2024 年時点で β 機能。プロダクション利用には enable が必要、UX も完全ではない
- **改善提案**: FR-005 を**スコープアウト**にするか、「Phase 2 リリース後の追加機能」として遅延させる。最初の Wave 1 では新規/既存判定して**新規はプロバイダ選択 / 既存はメールログイン**で割り切る

## 2. 曖昧すぎる要件（Phase 3 設計前に確定が必要）

### 🟡 Issue G-003: FR-007 「テキストモデレーション」の AI 判定方法が未指定
- **問題**: 「NGワード + AI判定」の AI 判定は具体的に何を使うのか？
- **質問**:
  - OpenAI Moderation API（無料・英語強い・日本語そこそこ）？
  - Google Perspective API（日本語サポート限定的）？
  - 自前で日本語特化モデル？
- **改善提案**: Phase 3 で**OpenAI Moderation を第一候補**にして比較表を作る（コスト無料、Edge Function から呼びやすい、日本語精度は手動テストで検証）

### 🟡 Issue G-004: FR-008 のチェックイン距離 GPS 検証で PostGIS 使用可否が未決定
- **問題**: NFR-006 の「100m 距離許容」をサーバー側 RPC で計算する場合、PostGIS の `ST_DWithin` を使うか、`earth_distance` モジュールで済ますかで実装が変わる
- **質問**: Supabase は標準で PostGIS を有効化できるが、既存のテーブルに座標カラムが geometry 型か lat/lon の数値型かで設計が変わる
- **改善提案**: Phase 3 設計書で**現状の facilities テーブルのカラム型を確認**してから決める。新規拡張を入れたくないなら earth_distance で十分（誤差は数十メートル）

### 🟡 Issue G-005: FR-012 「同一ユーザー連続レビュー制限」の N時間が未定義
- **問題**: 「N時間以内」の N が空欄
- **改善提案**: 24時間 を初期値に。施設訪問の頻度を考えると過剰拒否にならない。あとから設定可能にしておく

### 🟡 Issue G-006: FR-013 「管理者削除権限」の管理者ロール定義が未指定
- **問題**: そもそも「管理者」とは誰？ Supabase の RLS で `auth.users.is_admin` カラムがあるのか、別テーブル `admin_users` を持つのか
- **改善提案**: Phase 3 で `app_admins` テーブル（user_id PK）を新設し、RLS ポリシーで `EXISTS (SELECT 1 FROM app_admins WHERE user_id = auth.uid())` 形式で判定する。1人開発なので最初の管理者は自分のユーザーIDを直接INSERT

### 🟡 Issue G-007: NFR-016 「通報後の管理者通知」の手段が未確定
- **問題**: メール / Slack / Supabase ダッシュボード のどれにするか
- **改善提案**: 1人開発前提なら **Supabase ダッシュボード（reports テーブル + 管理者画面の最小実装）** が最軽量。メール/Slack は通報数が増えてからで OK

### 🟡 Issue G-008: NFR-004 「拒否方式」のフォールバック UX が曖昧
- **問題**: 障害時に「現在モデレーション機能停止中です」と出すが、ユーザーは何分後にリトライすればよいか不明。永続的にロックされるなら離脱率が上がる
- **改善提案**: タイムアウト時は「**5分後にリトライしてください**」と表示し、Edge Function 側でステータス（API稼働中/障害中）を expose する health-check エンドポイントを用意。クライアントは投稿前に health チェック → 障害なら投稿ボタン disable

→ G-002 と同じ理由でスコープアウト推奨

- **疑問**: 既にセッション32 で色分け確認済み。Wave 3 で改めて触る必要があるか？
- **改善提案**: 既存実装の現状を確認してから優先度を決める。問題なければ削除

## 4. 隠れた依存関係

### 🔴 Dep G-101: Apple Developer Program の加入確認は Wave 1 開始の絶対前提
- Apple Sign in は Apple Developer Program（年 $99）への加入が必須
- 未加入ならまず加入手続きが必要（数日〜1週間）
- **対応**: Phase 3 設計開始前に**ユーザーに加入状況を確認する**

### 🔴 Dep G-102: Google OAuth は GCP プロジェクトと OAuth Consent Screen 設定が必要
- iOS は GoogleService-Info.plist、Android は google-services.json が必要
- OAuth Consent Screen で「テスト中」だと特定の Google アカウントしかログインできない（公開申請が必要、数日かかる）
- **対応**: 同じく Phase 3 開始前に GCP プロジェクトの有無確認

### 🟡 Dep G-103: モデレーション API のキー管理は Supabase Secrets で
- Edge Function から呼ぶ API キーは絶対にクライアントに渡さない
- **対応**: Phase 3 設計時に「Secrets 設定 → Edge Function → API 呼び出し」の経路を図示する

## 5. その他細かい改善提案

- **NFR-017** の「2〜4週間」は Wave ごとに分解した方が現実的（Wave 1: 1週間 / Wave 2: 1〜2週間 / Wave 3: 1週間 / Wave 4: 数日）
- **AT-031（ログアウト）** で「再ログインで元状態復帰」とあるが、状態（お気に入り・チェックイン履歴）は DB 側にあるので自動復帰する。明示は不要
- **AT-032（アカウント削除）** は GDPR 配慮で「削除済みユーザーの投稿はどうなるか」の方針が必要（匿名化 / 一括削除 / 残存）
